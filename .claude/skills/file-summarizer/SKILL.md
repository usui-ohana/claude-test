# File Summarizer

指定されたファイルを読み込み、決まった3部構成（タイトル・トピック3つ・結論）で要約を出力するスキル。

## なぜこの形式か

要約の構成を毎回揺らがせると、複数ファイルを後で見比べたり、要約だけを抜き出して別の用途に使ったりするときに扱いにくくなる。タイトル・トピック3つ・結論という固定の骨格にすることで、出力の長さや切り口が違っても同じ型に揃い、後工程（一覧化、比較、転記など）がしやすくなる。

## 手順

1. **ファイルを読み込む**
   - テキスト/Markdown/コードファイルは Read ツールで直接読む。
   - PDFは Read ツールがそのまま開けるので、必要なページ範囲を指定して読む（ページ数が多い場合は全体を把握できる範囲で区切って読む）。
   - **ファイルが存在しない場合**: Read が失敗したら、推測で要約を作らず即座に処理を止める。指定されたパスを示した上で「ファイルが見つかりません」と伝え、似た名前のファイルが近くにあれば Glob などで確認して候補を提示する。パスのタイプミスや拡張子違いの可能性も一言添える。
   - **ファイルが極端に大きい場合**: 全体を読み切れない量であれば、読める範囲で要約するのではなく、まず分量をユーザーに伝え、要約対象を絞る（章・ページ範囲など）か全体を分割して扱うかを確認する。

2. **内容を理解する**
   - ファイル全体の主題・流れ・主張を把握する。断片的な引用の寄せ集めにならないよう、何が書かれたファイルなのか自分の言葉で説明できる状態にしてから次に進む。

3. **トピックは内容を代表する3つに絞る**
   - ファイル内で実際に最も重要度・分量が大きい3つの論点を選ぶ。無理に均等な3つを捻り出すのではなく、自然に立つ3本の柱を探す。論点が3つに収まらないほど多岐にわたる場合は、関連するものを束ねて3つに集約する。
   - 各トピックは1〜3文程度で、何が書かれているかが分かる具体性を持たせる（見出し語だけの羅列にしない）。指定された文字数制約がある場合は、トピックの説明文の長さをそれに合わせて調整する（下記「文字数の指定」参照）。

4. **結論は要約ではなく着地点を書く**
   - 「結論」はファイル全体のトピックの再掲ではなく、そのファイルが最終的に何を言おうとしているか、読み手が持ち帰るべき要点・示唆・次のアクションを書く。

5. **改善提案を作成する**
   - 要約を終えたら、ファイルの内容に対する具体的な改善提案を2〜4個書く。「もっと詳しく書くべき」のような一般論ではなく、そのファイル特有の不足点・矛盾・読みにくさ・次に足すと良い情報などを指摘する。
   - ファイルの性質によって視点を変える。例えばコードなら可読性・エラー処理・命名、ドキュメントなら構成・抜け漏れ・読み手にとっての分かりやすさ、企画書なら根拠・具体性・次のアクションの明確さ、といった観点で考える。
   - 改善提案がほぼ無いほど完成度が高いファイルの場合は、その旨を素直に書いてよい（無理に問題点を捻り出さない）。

## 出力形式

必ず以下のMarkdown見出し形式で出力する（見出しレベルは変えない）：

```markdown
# [ファイルの内容を表すタイトル]

## トピック1: [短い見出し]
[説明]

## トピック2: [短い見出し]
[説明]

## トピック3: [短い見出し]
[説明]

## 結論
[ファイル全体の着地点・要点]

## 改善提案
- [提案1]
- [提案2]
- [提案3]
```

- タイトルはファイル名そのものではなく、内容を反映したタイトルを自分で付ける。
- 元のファイルの言語（日本語/英語など）に合わせて要約を書く。
- 複数ファイルを要約する場合は、上記の形式をファイルごとに繰り返し、ファイル名を含む小見出しで区切る。

## 文字数の指定

ユーザーが「200字で」「全体で500文字くらいに」のように文字数を指定した場合、それは**要約全体（タイトル〜結論まで、改善提案は除く）のおおよその合計文字数**として扱う。

- 指定がない場合は、各トピックの説明1〜3文という標準の長さで書く。
- 文字数が指定された場合は、その制約に収まるようにトピックと結論の説明文の長さを調整する。極端に短い指定（例: 50字）の場合は、見出し構成自体は保ったまま各説明を1文程度に圧縮する。
- 指定文字数に対して元ファイルの内容量が多すぎる場合は、無理に切り詰めて情報を欠落させるのではなく、その旨を一言添えてから可能な範囲で要約する。
- 改善提案の文字数は指定の対象外とし、簡潔に保つ。
