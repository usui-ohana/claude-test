# Claude CodeのSubAgents・Skills・Commandsとは？違いと使い分けを徹底解説

Claude Codeを使い始めると、SubAgents（サブエージェント）、Skills（スキル）、Commands（コマンド）という3つの機能に出会います。名前は似ていますが役割はまったく異なり、正しく使い分けることで開発効率が大きく変わります。本記事では、それぞれの仕組みと違い、実務での活用方法を整理して解説します。

## Claude Codeにおける3つの拡張機能の概要

Claude Codeは単体でも高機能なAIコーディングツールですが、SubAgents・Skills・Commandsという3つの仕組みを使うことで、特定のプロジェクトやチームのワークフローに合わせて柔軟に拡張できます。まずは三者の立ち位置を簡単に整理します。

- **Commands（コマンド）**: `/コマンド名` で呼び出す、定型処理のショートカット
- **Skills（スキル）**: 特定タスクの「やり方」をまとめたナレッジパッケージ
- **SubAgents（サブエージェント）**: 独立したコンテキストで動く専門担当のAI

この3つは互いに排他的ではなく、組み合わせて使うことを前提に設計されています。実際、後述するようにSubAgentsはSkillsを取り込んで動くこともできます。

## Commandsとは何か

CommandsはClaude Codeに対して `/review` や `/commit` のように短いコマンドを打つことで、決まった処理を即座に実行させる仕組みです。Markdownファイル1枚で定義でき、内容はプロンプトのテンプレートそのものです。

Commandsが向いているのは、毎回同じ言い回しで頼んでいるような単純な定型タスクです。例えば「このdiffをレビューして」「コミットメッセージを作って」といった指示を、毎回フルで書くのは手間がかかります。これをコマンド化しておけば、`/code-review` と打つだけで同じ処理が走ります。

Commandsの特徴は、起動コストが低く、仕組みもシンプルであることです。複雑な分岐や外部ファイルの読み込みを必要としない、ワンショットの指示に向いています。逆に、状況に応じて参照すべき資料が変わったり、複数ステップの判断が必要な処理には不向きです。

## Skillsとは何か

Skillsは、特定のタスクをどう遂行するかという「やり方」そのものをパッケージ化した仕組みです。`SKILL.md` というファイルに、いつ使うべきか（description）と、どう実行するか（手順）を記述し、必要に応じてスクリプトやテンプレート、参考資料を同じフォルダにまとめて配置します。

Skillsの最大の特徴は、Claude自身が状況に応じて「このスキルを使うべきか」を判断して自律的に呼び出す点です。ユーザーが明示的にスキル名を指定しなくても、descriptionに書かれたトリガー条件に合致すれば自動的に発動します。例えば「文字数を数えて」という依頼に対し、文字数カウント専用のスキルが自動的に呼ばれる、といった動きです。

Skillsはさらに、スクリプト・テンプレート・参考ドキュメントといった複数ファイルを束ねて持てる点も強みです。SKILL.md本体は軽量に保ちつつ、詳細な仕様や出力フォーマットを別ファイルに分離し、必要なときだけ読み込む「progressive disclosure（段階的開示）」という設計が推奨されています。これにより、毎回のコンテキストを圧迫せずに、複雑なノウハウを保持できます。

Skillsが向いているのは、手順が決まっていて再現性が求められるタスクです。文字数カウントのような機械的な処理、特定フォーマットへの変換、社内ルールに従ったドキュメント生成などが典型例です。

## SubAgentsとは何か

SubAgentsは、独立したコンテキストウィンドウを持つ「専門担当」のAIです。メインのClaude Codeとのやり取りとは別に、特定の役割に特化した振る舞い・使用可能なツール・使用モデルなどを個別に設定できます。

SubAgentsの大きなメリットは、メインの会話のコンテキストを圧迫せずに、調査や検証といった重いタスクを任せられることです。例えば、コードレビュー専門のサブエージェントや、テスト実行専門のサブエージェントを用意しておけば、メインの開発フローを止めずに並行して作業させることができます。

SubAgentsはフロントマターで多くの設定が可能です。使用できるツールの制限（`tools`）、使用するモデルの指定（`model`）、権限モード、さらには起動時に読み込むSkillsの指定（`skills`）まで細かく制御できます。`skills` フィールドにスキル名を列挙すると、そのスキルの内容がサブエージェントの起動時にあらかじめコンテキストへ注入され、専門知識を持った状態でタスクに取り組めます。

SubAgentsが向いているのは、明確に切り分けられる専門領域のタスクです。ドキュメント執筆専任、テスト専任、レビュー専任のように役割を分割し、それぞれに適したツールやモデルを割り当てることで、全体のワークフローが整理されます。

## 3つの違いと使い分けの基準

ここまでの内容を整理すると、三者の違いは次のように要約できます。

Commandsは「決まった指示を一発で呼び出すショートカット」、Skillsは「特定タスクのやり方そのもの」、SubAgentsは「独立した専門担当」です。粒度で言えば、Commandsが最も軽量で、Skillsはやや重く手順とリソースを持ち、SubAgentsはさらに大きな単位で役割とコンテキストそのものを分離します。

実務での使い分けの基準は、おおむね次のように考えるとわかりやすいです。単純な定型指示で済むならCommands、特定の処理ロジックや出力フォーマットを再利用したいならSkills、コンテキストや権限・モデルを丸ごと分離して専門担当を立てたいならSubAgentsを選びます。

## 組み合わせて使うとさらに強力になる

重要なのは、これらは排他的な選択肢ではなく組み合わせて使う設計だという点です。例えば、SubAgentsのフロントマターでSkillsを指定しておけば、その専門エージェントは起動した瞬間から該当スキルのノウハウを持った状態で動き出せます。SEO記事執筆専門のサブエージェントに文字数カウント用のスキルを組み込んでおけば、執筆と文字数調整を一つの専門エージェントに任せられる、といった構成が可能です。

また、Commandsから複雑な処理を呼び出したい場合に、その内部でSkillsやSubAgentsを利用する設計にしておくこともできます。ユーザー側からは `/コマンド名` という単純な入力に見えても、裏側では適切なスキルや専門エージェントが動いている、という構成です。

## 具体的なファイル構成で見る違い

実際の定義方法を見ると、三者の違いがより明確になります。

Commandsは `.claude/commands/` 配下に1つのMarkdownファイルを置くだけで動きます。内容はそのままプロンプトとして展開されるテンプレートで、引数を差し込むこともできます。設定項目は少なく、思いついた定型作業を即コマンド化できる手軽さが魅力です。

Skillsは `.claude/skills/<スキル名>/` というフォルダ単位で構成します。中心は `SKILL.md` で、フロントマターに `name` と `description` を書き、本文に実行手順を記述します。同じフォルダ内に `scripts/`（決定的な処理コード）、`references/`（詳細仕様などの参考資料）、`assets/`（出力テンプレート等の素材）も置けます。これにより本体は短く保ちながら、必要な情報だけを必要なタイミングで読み込めます。文字数カウントのスキルなら、判定ルールを`references/`に、計測ロジックを`scripts/`に、出力フォーマットを`assets/`に分けて持たせる形です。

SubAgentsは `.claude/agents/<エージェント名>.md` という1ファイルで定義しますが、フロントマターの設定項目はCommandsより格段に多くなります。利用可能なツールの絞り込み、使用モデル、権限モード、最大ターン数、起動時に注入するSkillsの指定など、独立した担当者としての振る舞いを細かくチューニングできます。本文には、その担当者専用のシステムプロンプトとして役割や進め方を書き込みます。

## 実際の活用シーン

それぞれの仕組みがどのような場面で役立つのか、具体的なシーンで考えてみます。

Commandsが活躍するのは、チームで頻繁に繰り返す軽い作業です。「コミット前のdiffレビュー」「リリースノートの下書き」「決まったフォーマットのPR作成」のように毎回ほぼ同じ指示を出す作業はコマンド化の好対象で、メンバー全員が同じ品質の指示を毎回打ち込む手間がなくなります。

Skillsが活躍するのは、判定基準や出力フォーマットが厳密に決まっている作業です。「全角・半角・タブを区別して4パターンの文字数を数える」というルールが決まっているなら、毎回自然言語で説明し直すのではなくスキルとして固定すれば、判断のブレなく同じ精度で実行されます。スクリプトを同梱できるため、目視や暗算では誤りやすい処理を確実にこなせる点も強みです。

SubAgentsが活躍するのは、役割そのものを切り出したいケースです。コードレビュー専任、テスト実行専任、ドキュメント執筆専任のように担当を分けると、メインの会話のコンテキストを専門外の話題で消費せずに済みます。さらにSEO記事執筆専任のサブエージェントに文字数カウントのSkillを組み込めば、目標文字数になるよう自律的に本文を拡張・圧縮する複合的なワークフローも1つの担当者に任せられます。

## 導入の進め方

初めてこれらの仕組みを導入する場合は、段階的に進めるのがおすすめです。

まず、日々のやり取りの中で「毎回同じ指示を書いている」と感じる場面を見つけ、それをCommandsに切り出します。次に、その指示の中に「判定基準やフォーマットが厳密に決まっている部分」があれば、それをSkillsとして独立させます。最後に、複数の指示や判断をまとめて任せられる役割が見えてきたら、その役割をSubAgentsとして切り出し、必要なSkillsをフロントマターで紐づけます。

この順番で進めると、いきなり複雑なSubAgentsを設計するよりも無理がなく、各仕組みの役割の違いも自然に体感できます。

## よくある疑問

**SkillsとSubAgentsはどちらを先に作るべきか。** 多くの場合、先にSkillsとして手順を固めておき、後から複数のSkillsを束ねる担当者としてSubAgentsを作る順番がスムーズです。手順が固まっていない段階でSubAgentsを作ると、役割の境界が曖昧になりやすいためです。

**1つのSubAgentに複数のSkillsを持たせてよいか。** 問題ありません。フロントマターの `skills` フィールドには複数のスキル名を列挙できます。役割を遂行するために複数の専門知識が必要なら、すべて列挙して起動時に読み込ませてかまいません。

**Commandsの中でSubAgentsやSkillsを呼び出せるか。** 可能です。ユーザーから見えるインターフェースはコマンド1つでも、内部的にはその処理の中で専門のサブエージェントやスキルを利用する設計にできます。利用者の操作を簡単にしたまま、裏側の処理を高度化できる構成です。

## まとめ

Claude CodeのSubAgents・Skills・Commandsは、それぞれ異なる粒度でワークフローを拡張する仕組みです。Commandsは定型指示のショートカット、Skillsは再利用可能な手順とリソースのパッケージ、SubAgentsは独立したコンテキストを持つ専門担当のAIです。プロジェクトの規模やチームの運用に応じて、この3つを適切に組み合わせることで、Claude Codeをより自分たちのワークフローに合った形にカスタマイズできます。まずは身近な定型作業をCommands化し、再現性が必要な処理をSkillsに切り出し、役割が明確に分かれる領域をSubAgentsとして独立させる、という順番で導入を検討するとスムーズです。
